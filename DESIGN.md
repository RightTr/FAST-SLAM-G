# FAST-LIO-SAM + Nav2 2D Map Design

## 1. 问题定义

当前系统里实际存在两套不同来源的“地图”：

- `FAST-LIO-SAM` 产生的 3D 点云地图和位姿真值
- `slam_toolbox` 产生的 2D 栅格地图

这两套地图不一致并不奇怪，因为它们背后的优化对象和坐标链路不同：

- `FAST-LIO-SAM` 自己维护 LiDAR-Inertial 的位姿估计与图优化
- `slam_toolbox` 会基于 2D 激光数据单独维护一套 `map -> odom`

因此，即使两者都看起来在“建图”，本质上也是两个 SLAM 系统并行运行。只要它们各自独立优化，地图就一定可能错位。


## 2. 当前仓库状态

这个仓库不是纯 FAST-LIO 前端，已经带有一个 LIO-SAM 风格的后端：

- 前端：FAST-LIO2 高频里程计
- 后端：GTSAM 因子图优化、回环约束、关键帧管理

但当前实现里有一个对接 Nav2 不友好的点：

- 代码实际发布的是 `camera_init -> body`
- 并没有严格按配置项中的 `mapFrame / odometryFrame / baselinkFrame` 组织 TF

这意味着它更像“单全局系里直接发布位姿”，而不是标准导航框架常见的：

- `map -> odom -> base_link`

这会让 Nav2、costmap、AMCL/Localization 类组件更难直接接入。


## 3. 目标

目标不是再叠加一个新的 SLAM 后端，而是让系统只有一套全局真值：

- 由 `FAST-LIO-SAM` 负责 3D 建图与定位
- 从它的优化结果中派生出 2D 栅格地图
- 将这个 2D 栅格地图提供给 Nav2 使用

这样可以保证：

- 2D 地图和 3D 点云地图来自同一套位姿
- 闭环后两者能一起收敛
- 避免 `slam_toolbox` 和 LiDAR-IMU SLAM 各自维护一套地图


## 4. 推荐总体方案

推荐结构如下：

1. `FAST-LIO-SAM` 作为唯一定位/建图源
2. 使用优化后的全局点云或关键帧点云生成 2D 栅格图
3. 向 Nav2 提供：
   - 标准 TF：`map -> odom -> base_link`
   - `nav_msgs/OccupancyGrid`
   - 必要时提供局部障碍物点云给 local costmap

建议不要让 `slam_toolbox` 在在线导航时继续做 SLAM。

如果还保留 `slam_toolbox`，它只能作为：

- 离线制图工具
- 单独实验对比工具

而不能同时作为在线导航里的主地图源。


## 5. 2D 地图生成思路

### 5.1 输入源

优先使用下面两类输入之一：

- 后端优化后的全局点云
- 后端优化后的关键帧位姿 + 关键帧点云

不建议直接用“未经后端校正的原始 odom + 当前扫描”做长期 2D 主地图，因为这样虽然 2D/3D 短期一致，但会一起漂移。

### 5.2 投影流程

2D 栅格图生成可以按以下流程进行：

1. 接收优化后的全局点云
2. 按 `z` 范围裁剪
3. 可选做地面分离或高度分层
4. 将保留点投影到 XY 平面
5. 按分辨率写入 occupancy grid
6. 对未知、空闲、占用区域做后处理

### 5.3 关键参数

常见需要配置的参数：

- 地图分辨率，如 `0.05m` 或 `0.1m`
- `z_min / z_max`
- 是否只保留障碍物，不保留地面
- 栅格膨胀半径
- 更新频率
- 地图原点和尺寸

### 5.4 两类可选策略

策略 A：直接从全局点云生成静态 2D 地图

- 优点：实现简单，和 3D 地图天然一致
- 缺点：动态障碍物处理较弱，更适合先验地图或半静态环境

策略 B：全局 2D 地图 + 局部实时障碍层

- 全局层来自 FAST-LIO-SAM 投影结果
- 局部层来自实时点云或深度/激光障碍物观测

这是更适合 Nav2 的结构。


## 6. TF 设计建议

要接 Nav2，建议把 TF 链路规范化。

### 6.1 推荐链路

- `map -> odom`
- `odom -> base_link`
- `base_link -> lidar`

其中：

- `map` 表示全局一致地图系
- `odom` 表示局部连续、短时平滑的里程计系
- `base_link` 表示机器人本体系

### 6.2 在本仓库中的映射关系

当前代码中的：

- `camera_init` 更接近“全局参考系”
- `body` 更接近 `base_link`

因此初步可做如下映射：

- `camera_init` -> `map`
- `body` -> `base_link`

后续再补出标准的 `odom` 层。

### 6.3 两种落地方式

方式 1：简化接入

- 先把 `camera_init` 直接当 `map`
- 暂时令 `map == odom`
- 发布 `map -> base_link`

优点：

- 改动小，便于先跑通

缺点：

- 不完全符合 Nav2 最理想的 TF 结构

方式 2：标准接入

- 后端优化结果作为 `map`
- FAST-LIO 前端连续里程计作为 `odom`
- 用两者差值计算 `map -> odom`

优点：

- 符合标准导航架构
- 闭环和局部连续性可以兼顾

缺点：

- 需要更仔细地拆分前端位姿和后端校正位姿

从工程角度，建议先做方式 1 验证链路，再演进到方式 2。


## 7. 代码改造建议

建议拆成两个阶段。

### 阶段 1：统一坐标语义

目标：

- 消除代码里硬编码的 `camera_init/body`
- 真正使用 yaml 里的 frame 配置

需要修改的方向：

- 将 odom、path、point cloud、global map 的 frame_id 改为可配置
- 明确 `mapFrame`、`odometryFrame`、`baselinkFrame`、`lidarFrame` 的职责
- 检查所有 TF 发布点，避免混用历史命名

阶段 1 的目标不是重构算法，而是先让整套系统在 ROS 语义上是自洽的。

### 阶段 2：增加 2D 投影节点

新增一个 ROS2 节点，例如：

- `cloud_to_occupancy_node`

输入可以是：

- `/lio_sam/mapping/cloud_global`

输出可以是：

- `/map` (`nav_msgs/OccupancyGrid`)
- `/map_metadata`

可选附加输出：

- 投影后的 `PointCloud2`
- 调试图层

内部模块建议：

- 参数读取
- 点云订阅
- z 裁剪
- XY 投影
- 栅格更新
- OccupancyGrid 发布


## 8. 关于“挂一个后端”的判断

如果你的意思是：

- 给 FastLIO 增加图优化/回环/全局一致性模块

那么这个仓库已经基本具备了，不是当前的主要短板。

如果你的意思是：

- 增加一个“把 3D 位姿和点云压成 2D 栅格地图”的模块

那么这是值得做的，而且正是当前最合理的方案。

所以严格来说，不建议“再挂一个新的 SLAM 后端”，而是建议“增加一个 2D 地图派生层”。


## 9. Nav2 接入建议

推荐最终结构：

1. `FAST-LIO-SAM` 提供全局定位和 3D 地图
2. `cloud_to_occupancy_node` 提供全局 2D 栅格图
3. Nav2 global costmap 使用该 2D 地图
4. Nav2 local costmap 使用实时障碍物观测

如果机器人运行环境是强 3D 场景，还需要额外考虑：

- 坡道和楼层是否能被简单 2D 投影表达
- 低矮障碍和悬空障碍如何分类
- 动态点是否进入全局静态地图


## 10. 实施优先级

建议按这个顺序推进：

1. 统一 TF 和 frame 语义
2. 去掉在线 `slam_toolbox` 地图主导地位
3. 先做一个最小可用的点云投影 2D 地图节点
4. 在 RViz 中验证 2D 栅格与 3D 点云一致
5. 接入 Nav2 global costmap
6. 再根据需要补标准 `map -> odom` 分层


## 11. 最小可行版本

一个能尽快验证的 MVP 可以是：

- 保留当前 FAST-LIO-SAM 后端
- 把 `camera_init` 视作 `map`
- 从全局点云生成一张在线 `OccupancyGrid`
- Nav2 先消费这张地图进行全局规划

这个版本的意义在于先证明：

- 2D/3D 地图是否能一致
- Nav2 是否能稳定消费该地图
- 真正的障碍表达是否满足你的场景

如果这一步验证通过，再继续做标准 TF 分层和局部代价地图优化，会更稳。


## 12. 结论

结论很明确：

- 可以做
- 而且方向正确
- 但不建议再引入另一个独立 SLAM 后端

正确做法是：

- 让 `FAST-LIO-SAM` 成为唯一全局位姿来源
- 基于其后端优化结果生成 2D 栅格地图
- 让 Nav2 使用这张由 3D 地图派生出的 2D 地图

这样才能从系统结构上保证一致性，而不是靠两套 SLAM 之间“对齐”。
